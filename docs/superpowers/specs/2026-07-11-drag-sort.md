# Spec：設定頁排序改版 — 拖拉排序＋序號直填＋新增指定順序（2026-07-11）

> **狀態：已完成（2026-07-11）**。含追加需求：各列「修改」鈕全拔，改為每區塊右上角一顆「儲存修改」（快照 diff，只送有變動列）。157 測試綠、瀏覽器實測通過。未 commit。

## 背景

- 設定頁四張維護卡（品項名稱 Category／廠牌 Brand／手機品牌 PhoneBrand／手機型號 PhoneModel）目前用「上移／下移」按鈕（`settings.js` 的 `_swapSort`）。
- **已知 bug**：匯入的型號 sort 值全部相同，交換相同值等於沒換 → 使用者按上下移沒反應。新做法儲存時整組重寫 1..N，順帶治好。
- 互動模式參考 `docs/drag_sort_demo.py`（PySide6 範例，只搬精神不搬 code）：
  1. 整列拖拉（⠿ 把手）
  2. 序號格點擊直接打數字＋Enter 搬到該位置
  3. 改動只動記憶體，亮「儲存排序」鈕，按下才寫 DB
- **使用者明確要求：抽成共用元件**，四張卡一起換，不要各寫各的。

## 一、前端共用元件 `sortable-list`

新檔 `static/js/sortable.js`，掛 `window.PosComponents["sortable-list"]`（inline template 字串）；`app.js` 仿 `PosPages` 迴圈註冊 `window.PosComponents`。`index.html` 加 `<script src="/js/sortable.js">`（放在 settings.js 前）。

Props / 事件：

```
<sortable-list :items="categories" item-key="category_id" @save="saveSort('categories', $event)">
  <template #row="{ item, index }"> ……原本那列的 input＋按鈕…… </template>
</sortable-list>
```

- `items`：陣列（prop 變更時 watcher 重置內部 rows 複本、清 dirty）。
- `item-key`：主鍵欄名。
- `@save`：回傳目前順序的 id 陣列 `rows.map(r => r[itemKey])`。

元件內部每列渲染：`⠿ 把手｜序號格｜<slot name="row">`，並：

- **拖拉**：`draggable` 放在 ⠿ 把手 span 上（不要放整列——列內有 input，會跟文字選取打架）；row 接 `@dragover.prevent`／`@drop`。drop 目標列即插入位置（demo 的 `RowDragFilter` 邏輯：`rows.insert(dst, rows.pop(src))`）。拖過的列給視覺提示（如上邊框變色）。
- **序號格**：顯示目前位置 `i+1`，可點擊輸入數字，Enter／blur 提交。驗證同 demo `parse_seq_move_target`：非數字或超出 1..N → 安靜跳回原數字；合法 → 搬移。用 `:value` + 事件寫回，不要 v-model（重繪時會打架）。
- **dirty 機制**：任何搬移後亮「儲存排序」＋「取消」鈕（列在清單底部）；儲存 emit `save` 後清 dirty；取消還原成 props 順序。
- 樣式加進 `pos.css`：`.drag-handle`（cursor:grab、灰色）、`.seq-cell`（寬 ~48px、置中、淺色虛線框提示可點）、拖拉目標列 highlight。

四張卡改法（`index.html` `#tpl-settings`）：

- 刪掉所有 上移／下移 按鈕與 `settings.js` 的 `_swapSort`／`moveCat`／`moveBrand`／`moveModel`。
- Category、Brand、PhoneBrand：整張清單一個 `<sortable-list>`。
- PhoneModel：**每個手機品牌群組一個** `<sortable-list>`（`modelGroups` computed 已分組；跨品牌不可拖）。

## 二、後端排序端點

`api/catalog.py` 加共用 helper＋四條路由（已確認與現有路由無衝突）：

```
PUT /api/categories/sort    body {"ids": [3,1,2]}
PUT /api/brands/sort
PUT /api/phone-brands/sort
PUT /api/models/sort
```

- 依 ids 順序重寫 `sort = 1..N`（單一交易）。
- 型號送的是「單一品牌群組內」的 ids，只重寫這些列；群組間靠 `pb.sort` 先排，組內 1..N 重複無妨（現有排序 `ORDER BY pb.sort, m.sort, m.model_id`）。
- 驗證：任一 id 查無 → 422，整筆不寫。
- pydantic：`class SortIds(BaseModel): ids: list[int]`。
- ⚠️ 路由註冊順序：`/models/sort` 之類的字面路徑需在 `patch /models/{mid}` 等參數路由**之前**註冊才保險（FastAPI 依註冊順序匹配；PUT vs PATCH 方法不同其實不撞，但放前面最穩）。

## 三、新增指定順序模式

四張卡的「新增」表單各加一格選填「序號」input（w60，placeholder「序號(選填)」）：

- 不填 → 照舊排最後（現有 `_next_sort`）。
- 填 n → 前端流程：POST 新增（拿回新 id）→ reload 清單 → 在對應清單（型號是該品牌群組）把新 id 移到第 n 位 → 呼叫同一支 `PUT …/sort`。**不要**在後端 POST 做位移特例，重用 sort 端點即可。
- n 超出範圍 → 夾到 1..N（不報錯）。
- `settings.js` 寫成共用 helper（如 `insertAtPosition(kind, list, newId, pos)`），四個 add 方法共用。

## 四、`settings.js` 新增

```js
async saveSort(kind, ids) {
  const m = _MAINT[kind];
  await API.put(m.url + "/sort", { ids });
  await this.reloadAll();
}
```

（models 群組傳該群組 ids 即可；`modelGroups` 是 computed，reload 後 prop 更新元件自動重置。）

## 五、測試（tests/，unittest）

- sort 端點：重排 categories 後 GET 順序符合；不存在的 id → 422 且原順序不變。
- models：同品牌群組重排後 `ORDER BY pb.sort, m.sort` 順序正確；不影響他品牌。
- 全部同 sort 值（重現原 bug 資料）→ 重寫後 1..N。
- `parse_seq_move_target` 等驗證邏輯在前端，不用 python 測。

## 六、驗收（瀏覽器）

1. 設定頁四張卡都看得到 ⠿ 與序號格，上移下移按鈕消失。
2. 拖一列 → 亮「儲存排序」→ 儲存 → 重新整理後順序保留。
3. 序號格打數字＋Enter 搬位；打非法值跳回。
4. iPhone 型號群組內拖拉可動（原 bug 情境）。
5. 新增時填序號 → 出現在指定位置。
6. 不儲存直接重新整理 → 順序還原。
